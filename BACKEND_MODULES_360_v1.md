## 0. STACK DECISIONS (firm unless CEO overrides)

| Layer | Choice | Why |
|---|---|---|
| Web app | Next.js 16 (App Router) | Apex stack standard; Maitro Phase 0 locked 16.2.4 |
| API workspace | Node 20 LTS + TypeScript + Fastify or tRPC | Familiar lane, low-overhead |
| DB | Postgres 16 (managed if budget allows; otherwise on-VPS with daily backup) | RDBMS for assessments/grading; JSON columns for question payloads |
| Cache / queue | Redis 7 (already on VPS) | Existing investment |
| Background workers | BullMQ on Redis | Standard, no new infra |
| AI grading | OpenRouter + Anthropic direct for premium grading; LiteLLM+Qwen3 (BHIMA tunnel) for bulk/low-stakes | Cost-route per CTO labor split; no Ollama on VPS |
| Code-execution sandbox | Judge0 self-hosted OR Piston (containerized) on BHIMA, exposed via Cloudflare tunnel | Off-VPS to protect main-box budget |
| Object storage | S3-compatible (Cloudflare R2 cheapest) | Video, attachments, evidence |
| Auth | NextAuth.js + email/OAuth + SAML/OIDC for enterprise (M18) | Standard, well-integrated with Next |
| Payments | Razorpay (India) + Stripe (international) | CVPRO `payment-gateway` reuse — see skill |
| Email | Resend or AWS SES | Transactional |
| Observability | OpenTelemetry → existing Talpro Sentinel + Pino logs → existing fleet | Reuse, do not duplicate |
| Deployment | Atomic-release pattern (build /tmp → releases/<SHA> → flip symlink → pm2 reload --update-env) | Lessons from Setu + Turnstile incidents |

Each new PM2 service must keep mem ≤ 150MB at idle to stay inside the 8GB VPS budget. Heavy AI workloads route to BHIMA via tunnel.

## 1. MODULE-BY-MODULE SPEC

Format per module: **ID** — Owner sub-lane — Exit criteria — Dependencies — Notes.

### M1.A — Assessment Builder
- **Outcome:** authenticated user can compose an assessment from a question palette (MCQ, multi-select, short-answer, code-in-browser, file upload, video response), reorder sections, set time limits, randomize, preview as candidate.
- **Exit:** create → preview → save → share via signed link → candidate can complete. Persists. Editable.
- **Depends on:** auth, DB schema, file storage.
- **Notes:** start with 4 question types (MCQ, multi-select, short-answer, code). Video and file-upload land in M6.

### M1.B — Assessment Library (seed)
- **Outcome:** 25-skill library populated with 10–20 calibrated questions per skill (from GAP_ANALYSIS Section E1 seed list).
- **Exit:** library browse page returns 25 skill packs; each pack can be cloned into a new assessment in one click.
- **Depends on:** M1.A schema, skill taxonomy (M12).
- **Notes:** seed content authored by SMEs or auto-generated by LLM with human review (CEO call). Track provenance in DB.

### M2 — Coding sandbox
- **Outcome:** in-browser code editor (Monaco) for ~15 languages (JS/TS/Python/Java/C++/C#/Go/Ruby/PHP/Kotlin/Swift/Rust/SQL + HTML/CSS preview); runs code in Judge0 or Piston sandbox on BHIMA tunnel; returns stdout, stderr, time, memory.
- **Exit:** Python+JS+Java solving a fizzbuzz problem end-to-end. Latency < 3s median.
- **Depends on:** BHIMA sandbox container live + Cloudflare tunnel.
- **Notes:** rate-limit per session. Hard memory/CPU caps in sandbox config.

### M3 — Job simulations
- **Outcome:** scenario question type — multi-step task replicating real role work (e.g., "triage these 5 support tickets," "build this spreadsheet model").
- **Exit:** 3 simulations live in library, AI-gradable.
- **Depends on:** M1.A, M4, M6 (for video-step simulations).
- **Notes:** product page promised in W6 only after this ships.

### M4 — AI grading engine v1
- **Outcome:** every non-objective answer (short-answer, code, video transcript) gets a graded score + reasoning trace + calibration interval. Two-layer grader: cheap-LLM first pass → premium-LLM second pass on top X% and bottom X% for sanity.
- **Exit:** 100 graded answers with audit-log row each, human-spot-check accuracy ≥ 85% vs SME rubric.
- **Depends on:** OpenRouter/Anthropic keys, M21 audit log, M14 IRT (later integration).
- **Notes:** **always return reasoning trace + confidence band.** Never a bare score. Bias audit (M16) hooks here.

### M5 — Cognitive / aptitude engine
- **Outcome:** numerical reasoning, verbal reasoning, abstract reasoning, situational judgment question types with adaptive difficulty.
- **Exit:** 4 cognitive packs in library; adaptive serving working (IRT M14 dependency).
- **Depends on:** M14.

### M6 — Video response + transcription scoring
- **Outcome:** candidate records video (≤2 min) in-browser → uploaded to R2 → transcribed (Whisper or Deepgram) → graded by M4.
- **Exit:** 5 video responses captured, transcribed, graded, evidence stored.
- **Depends on:** M4, R2 storage.

### M7 — AI screening chatbot (Conversational Agent)
- **Outcome:** asynchronous conversational pre-screening: candidate clicks a link, chats with bot through a structured Q-set, transcript scored by M4.
- **Exit:** 10 candidates screened end-to-end. Transcript + score per session in DB.
- **Depends on:** M4. Vervoe parity feature — needed for high-volume.

### M8 — Interview scheduling
- **Outcome:** scheduling link with calendar integration (Google Cal + Outlook) → candidate self-picks slot → automatic invites + reminders.
- **Exit:** book/reschedule/cancel flows clean; ICS files valid; reminders fire.
- **Depends on:** Google + Microsoft OAuth scopes.

### M9 — Live coding interview room
- **Outcome:** real-time coding room (multi-cursor, language server, run-button, take-home option). WebSocket-based.
- **Exit:** two browsers, live edit, run code, transcript saved.
- **Depends on:** M2 sandbox, WebSocket server.
- **Notes:** can defer to Phase 7 if CoderByte parity not urgent for early customers.

### M10 — Reference checking
- **Outcome:** structured async reference questionnaire sent to candidate's referees; responses aggregated into a report.
- **Exit:** 1 candidate, 3 referees, aggregated report produced.

### M11 — Anti-cheat + Gen-AI detection
- **Outcome:** browser-side signals (tab switching, paste detection, full-screen exit, copy from external), plus LLM-based answer-pattern detection ("looks AI-generated"), surfaced as a risk score per session.
- **Exit:** at least one realistic GPT-pasted answer flagged with confidence ≥ 0.7.
- **Depends on:** M4 (shares grader infra).

### M12 — Skill taxonomy
- **Outcome:** canonical skill graph: 150+ skills (TC catalog floor), hierarchies (skill → sub-skill → competency), tags (technical/behavioural/cognitive), role-mappings.
- **Exit:** every skill in library + every JD-Forge skill resolves to taxonomy node. JSON export at `/api/skills` for partners.
- **Depends on:** none — foundation.
- **Notes:** seed from TC `/skill-library` extract in GAP_ANALYSIS §C.

### M13 — JD-Forge (WEDGE)
- **Outcome:** paste a job description → extract skills with weights → suggest assessment composition (sections, question types, time, difficulty) → preview → ship.
- **Exit:** 10 JDs in, 10 calibrated assessments out in < 60s each, human-edit pass rate ≥ 70% (i.e., humans accept the draft with light edits).
- **Depends on:** M4 (LLM grader infra), M12 (taxonomy), M14 (calibration).
- **Notes:** this is the marketing wedge. Quality bar is high.

### M14 — IRT calibration engine (WEDGE)
- **Outcome:** background job that recomputes per-item difficulty (b), discrimination (a), and pseudo-guessing (c) using 3PL model when an item has ≥ 50 responses. Item stats surfaced in builder + library.
- **Exit:** 25 seed-library items with computed IRT params + reliability score. JSON export for tribunal-friendly report.
- **Depends on:** M1.A response logs.
- **Notes:** this is what makes QOrium psychometrically defensible. Cite published methodology on `/trust/bias-audit`.

### M15 — ISO 27001 + DPDP hardening
- **Outcome:** policies, controls, evidence repository, internal audit, gap remediation, ready for Stage 1 audit.
- **Exit:** Stage 1 ISO audit booked; DPDP self-assessment 100%.
- **Notes:** post-revenue. CEO chooses auditor.

### M16 — Independent bias audit (WEDGE)
- **Outcome:** third-party bias audit of M4 grader + M13 JD-Forge. Indian auditor first (e.g., NASSCOM-affiliated). Report published at `/trust/bias-audit`.
- **Exit:** auditor letter received, report published, schema marked up.

### M17 — India data residency
- **Outcome:** primary DB + R2 bucket in Mumbai region. Backup in Hyderabad/Chennai. Data-flow diagram on `/trust/security`.
- **Exit:** all candidate PII verified resident in India per audit log.

### M18 — SSO (SAML + OIDC)
- **Outcome:** enterprise customers authenticate via SAML (Okta, Azure AD, Google Workspace, JumpCloud) and OIDC.
- **Exit:** Okta tenant test, Google Workspace test, Azure AD test all green.

### M19 — ATS / HRIS integrations
- **Outcome:** integrations with the top 4 ATSes used in India (Greenhouse, Lever, Workday, Naukri RMS). Webhook + REST.
- **Exit:** create-candidate from ATS → assessment auto-sent → results push back to ATS. Tested with at least 2 ATS sandboxes.
- **Depends on:** M20.

### M20 — Public API + webhooks
- **Outcome:** REST API surface (`/api/v1`) covering assessments, candidates, results, skills. OpenAPI 3 spec. Bearer-token auth. Webhook signing (HMAC).
- **Exit:** SDK examples (Node, Python) in `/docs`. Postman collection. 100% endpoints have RFC 7807 errors.
- **Depends on:** M21 audit log on every mutating call.

### M21 — Audit log
- **Outcome:** append-only audit log of: grade decisions, assessment changes, user actions, permission changes, data exports. Exportable as CSV.
- **Exit:** every grade decision in M4 produces an audit row with input hash, model, score, reasoning hash, timestamp, user.
- **Notes:** mandatory from Phase 1 — do not defer.

### M22 — SOC 2 Type II
- **Outcome:** SOC 2 Type II report.
- **Exit:** Type II report letter.
- **Notes:** triggered after ₹1cr ARR or first enterprise design-partner request. Late-phase.

### M23 — Priority support tier + SLAs
- **Outcome:** enterprise customers get private Slack channel, P1 SLA (≤ 1h response), uptime SLA (99.9% credits).
- **Exit:** at least one enterprise contract signed with SLA addendum.

## 2. DATA MODEL — FIRST CUT (Codex to refine)

Core entities:
- `org` (tenant)
- `user` (with role: owner, admin, recruiter, viewer)
- `candidate` (PII separated, deletable per DPDP request)
- `skill` (taxonomy node)
- `question` (typed, versioned, has IRT params)
- `assessment` (composed of sections, sections of questions)
- `assessment_invite` (signed link, expiry)
- `attempt` (candidate session, contains answers)
- `answer` (per-question, has grade, reasoning_trace_ref, confidence)
- `grade_decision` (audit row referencing answer + model + version)
- `job_description` (for JD-Forge inputs)
- `jd_forge_run` (output of one Forge run linked back to assessment)
- `webhook_endpoint` (per-org)
- `webhook_delivery` (with retry state, HMAC signature)
- `audit_log` (immutable, append-only, partitioned by org_id+day)
- `subscription` (org's pricing tier, Razorpay/Stripe IDs)
- `usage_metric` (per-org per-month, for tier-limit enforcement)

All tables: `id (uuid v7)`, `org_id (fk)`, `created_at`, `updated_at`, `deleted_at` (soft-delete except audit_log).
Strict foreign keys. No `org_id` nulls.

## 3. NON-FUNCTIONAL REQUIREMENTS

| Area | Requirement |
|---|---|
| Performance | p50 API ≤ 200ms; p95 ≤ 800ms; p99 ≤ 2s |
| Reliability | 99.9% uptime on backend; queue lag ≤ 30s for grading |
| Security | TLS 1.3, HSTS, X-Content-Type-Options, X-Frame-Options, CSP, rate-limit 10r/s burst 20 on public routes, gitleaks in CI, no secrets in code |
| Privacy | DPDP-aligned consent capture, PII export within 30 days of request, PII erase within 30 days, sub-processor list public |
| Logging | Pino structured logs to fleet; PII never in logs |
| Errors | RFC 7807 Problem Details on every API error |
| Backups | Daily DB snapshot, 30-day retention, restore drill quarterly |
| Disaster recovery | RPO ≤ 24h, RTO ≤ 4h |
